## 1. 專案背景

本公司負責嘖嘖（Zeczec）募資代操案件，每檔皆有突破千萬募資金額的潛力。目前負責募資的僅有一人，若負擔過大將外包部分執行或增員，但所有案件仍以該人為主要掌控者。

因此需要一套專案管理系統，能自動排程、管理多案件時間軸、追蹤 KOL 合作進度，並在隨時有變動時快速調整。

### 1.1 案件概要（範例）

> 以下三檔案件僅為範例，系統應支援動態新增任意案件。

| 案件名稱 | 模板 | 排程模式 | 電檢狀態 | 備註 |
|---------|------|---------|---------|------|
| SABA RO 飲水機 | 全自操 | 正推 | 已完成 | 可立刻啟動，需自行拍攝商品照/影片 |
| BleeqUP AI眼鏡 | PM模式 | 反推 | 未通過 | 問卷已完成。不拍攝，用原廠素材。電檢預計 2-3 個月 |
| INMO AI眼鏡 | PM模式 | 反推 | 未通過 | 不拍攝，用原廠素材。與 BleeqUP 同廠商送檢 |

### 1.2 使用者工作參數

- 每日可用工時：8 小時
- 工作日：週一至週五（週末不工作）
- 已知不可用時段：6/20–6/26 出國（可能安排職代）、7/10–7/16 員工旅遊（全公司出國）
- 品牌端審稿一輪：抓 3 個工作天
- 設計師外包交稿：依任務 1–7 個工作天不等

## 2. 系統目標與核心功能

### 2.1 系統定位

本系統是一人管理多檔募資案的「排程大腦」，核心價值在於：自動依賴排程、跨案件產能衝突偵測、走期錨定、KOL 個別時程管理。

### 2.2 核心功能清單（第一版）

| # | 功能 | 說明 |
|---|------|------|
| F1 | 多案件甘特圖 | 所有專案並列的時間軸，有今日標記線、走期里程碑標記（問卷/開賣/結束） |
| F2 | 自動依賴排程引擎 | 基於任務依賴關係拓撲排序，考慮工時容量、等待天數、跨案件產能分配 |
| F3 | 雙向排程模式 | 正推（A 模式）：從啟動日推算最快上線時間。反推（B 模式）：從目標上線日反推啟動時間 |
| F4 | 走期設定 | 問卷上線/結束日、募資上線/結束日，排程引擎依走期錨定任務時間 |
| F5 | 任務模板與勾選 | 「全自操」與「PM 模式」兩種模板，建案時套用再微調。每項任務可勾選啟用/停用 |
| F6 | KOL 個別時程管理 | 每位 KOL 獨立里程碑（9 個節點），填入寄出日期自動推算，可個別調整 |
| F7 | 今日總覽 + 甘特圖雙檢視 | Dashboard 顯示今日任務清單與未來 7 天待辦，可切換至甘特圖檢視 |
| F8 | 持久儲存 | 資料跨 session 保存（目前使用 Claude persistent storage，正式版建議改用後端資料庫） |

**實作狀態：**

- [✅] F2 自動依賴排程引擎
- [✅] F3 雙向排程模式

- [✅] F5 任務模板與勾選
- [✅] F6 KOL 個別時程管理

## 3. 任務資料庫結構

系統內建約 60 項任務，涵蓋募資全流程 6 大階段。每項任務包含以下欄位：

| 欄位 | 型別 | 說明 |
|------|------|------|
| id | string | 任務編號（如 1.1、2.14、5.13） |
| name | string | 任務名稱 |
| phase | enum | 所屬階段：1A / 1B / 2 / 3 / 4 / 5 / 6 |
| hrs | number | 全自操模式預估工時（小時） |
| pmH | number | PM 模式預估工時（小時，拍攝類為 0） |
| deps | string[] | 前置依賴任務 ID 陣列 |
| wait | number | 完成後等待天數（如客戶審稿 3 天、設計師交稿 7 天） |
| dl (deadline) | enum | "sv" = 問卷上線前完成、"cp" = 募資上線前完成 |
| timing | enum | "dsv" = 問卷期間、"dcp" = 募資期間、"post" = 募資結束後 |
| shoot | boolean | 是否為拍攝相關任務（PM 模式自動停用） |
| span | number | 跨度天數（用於每日追蹤類持續性任務，如 21 天、45 天） |

**實作狀態：**

- [✅] 任務資料庫結構定義
- [✅] 約 60 項任務資料建置

### 3.1 六大階段與任務依賴邏輯

以下為任務之間的關鍵依賴路徑（詳細任務清單見附件 Excel 工作拆解）：

**關鍵路徑 1：網頁線（最長瓶頸）**

```
競品分析(1.2) → 網頁大綱(2.13) → 網頁初版(2.14) → 客戶審稿(+3d) →
網頁Final(2.15) → 設計前置會議(2.16) → 設計外包(+7d)(2.17) →
設計過稿(+3d)(2.18) → 長條圖建置(3.1) → 募資上線
```

**關鍵路徑 2：影片線（全自操模式）**

```
競品分析(1.2) → 影片企劃(2.1) → 客戶審稿(+3d)(2.2) → 拍攝前置會議(2.3) →
拍攝(2.4) → 影片過稿(+3d)(2.5) → 素材交付給網頁設計
```

**關鍵路徑 3：KOL 線**

```
競品分析(1.2) → KOL 名單(4.1) → 詢價(+5d)(4.2) → 敲定(4.3) → 簽約(4.4) →
Brief(4.5) → 腳本過稿(+5d)(4.6) → 影片過稿(+5d)(4.7) → B copy(4.8) → 上線授權(4.9)
```

> 三條路徑的匯合點：募資上線日。網頁 + 影片 + KOL 都必須在上線前到位。

**實作狀態：**

- [✅] 關鍵路徑 1：網頁線依賴關係
- [✅] 關鍵路徑 2：影片線依賴關係
- [✅] 關鍵路徑 3：KOL 線依賴關係

### 3.2 可平行執行的任務群組

> **注意：** 影片和照片素材需在問卷上線前取得。

- [✅] 合約簽訂後：時程表確認(1.3) / 預算規劃(1.4) / 產品定價(1.5) 可同步進行
- [✅] 企劃核心期：影片線 / 平面線 / 問卷線 / 網頁線 四條可平行推進
- [✅] 影片與平面拍攝可排在同一天
- [✅] 廣告帳號設定(5.1–5.5) 不依賴企劃內容，可提前處理
- [✅] KOL 名單搜集(4.1) 可在企劃階段早期就開始
- [✅] 網頁企劃大綱(2.13) 可在平面照未精修前開始撰寫

## 4. 排程引擎規格

### 4.1 正推模式（A 模式）

適用場景：產品已通過電檢，可立刻啟動的案件（如 SABA）。

- [✅] 使用者輸入「專案啟動日」
- [✅] 系統依任務依賴關係進行拓撲排序
- [✅] 由啟動日往後推算每項任務的開始/結束日
- [✅] 考量每日工時上限（8hr）、週末、不可用時段
- [✅] 標記為「問卷前」的任務完成後，系統推算「最快可問卷上線日」
- [✅] 標記為「開賣前」的任務完成後，系統推算「最快可募資上線日」
- [✅] 使用者確認後填入走期，排程自動校正

### 4.2 反推模式（B 模式）

適用場景：已知目標上線時間，需反推最晚啟動日的案件（如 BleeqUP、INMO）。

- [✅] 使用者輸入「目標募資上線日」（及可選的「問卷上線日」）
- [✅] 系統計算所有「開賣前」任務的總工時與等待天數
- [✅] 從問卷上線日或募資上線日往前反推，算出最晚啟動日
- [✅] 使用者可手動將啟動日設定得更早，增加緩衝時間
- [✅] 當啟動日早於系統建議日，多出的時間自動分散到各任務之間

### 4.3 走期錨定邏輯

| 走期欄位 | 影響的任務 | 排程行為 |
|---------|----------|---------|
| 問卷上線日 | deadline="sv" 的所有任務 | 這些任務必須在此日前完成 |
| 問卷結束日 | timing="dsv" 的任務 | 問卷追蹤等任務的結束上限 |
| 募資上線日 | deadline="cp" 的所有任務 | 這些任務必須在此日前完成 |
| 募資結束日 | timing="dcp" 的任務 | 每日追蹤/社群貼文的結束上限 |

**實作狀態：**

- [✅] 走期錨定邏輯實作

### 4.4 跨案件產能分配

- [✅] 所有專案共用同一個 dailyLoad 容量池
- [✅] 每日上限 = hoursPerDay（預設 8），跨案件任務合計不超過此上限
- [✅] 排程優先順序：所有專案的任務混入**全域任務池**，每次從「dependencies 已完成且 startDate 已到」的任務中，挑出優先度最高的排入
  - 主要排序鍵：專案啟動日越早 → 優先度越高
  - 次要排序鍵：關鍵路徑剩餘工時越長 → 優先度越高（長鏈任務不被短鏈擠佔）
  - 效果：後加入的專案只要啟動日合理，即可從自己的 startDate 起與其他專案公平競爭當日產能，不再因陣列順序而整體延後
- [✅] 週末與不可用時段（如出國、員工旅遊）自動跳過

> **未來升級方案 C（優先權動態調整）**：當有需求在「任務完成後即時更新後繼任務優先度」時，可將目前的 `schedulePool` while 迴圈（O(n²) 線性掃描）替換成 min-heap 驅動的事件模擬器（O(n log n)）。heap 的比較鍵與現行 `calcPriority` 公式相同，只是從「每輪重排」改為「任務完成時 push 後繼」。`isReady`、`calcPriority`、`allocateTask`、`reshapeResult` 四個純函式均不需修改。

## 6. KOL 個別時程管理

每檔案件可新增多位 KOL，每位 KOL 擁有獨立的里程碑時程。KOL 分兩波段上線：第一波至少 4 位在募資開賣當天上線，第二波為中期補強。

### 6.1 KOL 里程碑節點（單人，共 9 個）

| # | 里程碑 | 預設偏移 | 說明 |
|---|--------|---------|------|
| 1 | 寄出產品 | Day 0 | 基準日，填入後自動推算其餘 |
| 2 | 提供拍攝 Brief | Day +1 | 品牌方提供 |
| 3 | 創作者提供腳本初版 | Day +6 | 創作者提供 |
| 4 | 品牌腳本回饋 | Day +8 | 至少 2 個工作天後回覆 |
| 5 | 腳本確認完成 | Day +9 | |
| 6 | 創作者提供影片 A copy | Day +13 | 創作者提供 |
| 7 | 品牌影片回饋 | Day +15 | 至少 2 個工作天後回覆 |
| 8 | 影片＋文案 Final | Day +16 | |
| 9 | 上線 | Day +17 | 需在募資上線日或之前 |

**實作狀態：**

- [✅] KOL 里程碑節點（9 個）資料結構
- [✅] 波段設定（第一波/第二波）

### 6.2 自動推算邏輯

- [✅] 使用者填入「寄出產品」日期後，其餘 8 個節點依預設偏移天數自動填入
- [✅] 每個節點可個別手動調整，調整後不影響其他節點
- [✅] KOL 里程碑會出現在今日總覽的待辦清單中

## 9. 技術建議

### 9.1 建議技術棧

- [✅] 前端：React（函式元件 + Hooks）
- [✅] 狀態管理：useState / useReducer + Context，或 Zustand
- [✅] 持久化：後端 API + 資料庫（PostgreSQL / SQLite），或 localStorage 做 MVP
- [] 行事曆匯出（V2）：ics.js 生成 .ics 檔
- 不建議串 Notion：自動排程邏輯超出 Notion 能力，維護成本高

### 9.2 排程引擎實作要點

- [✅] 拓撲排序（Topological Sort）解決任務依賴（per-project DFS，depKeys 以 `${pid}::${id}` 命名空間隔離跨案件）
- [✅] 全域任務池排程（Global Task Pool Scheduling）：
  - 所有專案任務展平成單一 PoolEntry 陣列
  - 主迴圈：每輪從「ready（前置完成 + startDate 已到）」任務中以 `calcPriority` 排序，取優先度最高者呼叫 `allocateTask`
  - `calcPriority = projStart.getTime() × 1e⁻⁶ − remainH`（啟動日為主鍵，關鍵路徑長度為次鍵）
  - 取代舊的「逐專案串行排程」，讓晚啟動的案件從自己的 startDate 起與其他案件公平競爭
- [✅] 容量控制：每日累計工時不超過 hoursPerDay，跨所有專案共用同一 load map
- [✅] 等待天數：任務完成後 +N 個工作天才觸發下一個依賴（waitEnd = aWD(tE, w, bl)）
- [✅] 反推模式：先估算總工時 + 等待天數，從目標日期往前扣除
- [✅] 安全迴圈上限：while 迴圈加 safety counter 防無限迴圈（任務池：`pool.length² + 10`；單任務分配：500）
- [] **未來升級方案 C**：將 `schedulePool` 的 while 迴圈替換為 min-heap 事件模擬器，支援任務完成後即時動態調整後繼優先度；其餘四個純函式（`isReady`、`calcPriority`、`allocateTask`、`reshapeResult`）介面不變

### 9.3 注意事項

- 跨案件衝突是最大挑戰：三檔案件的企劃核心期（W3–W4）和上線前衝刺期最容易撞車
- 三檔案件的上線時間建議至少間隔 4–6 週，否則企劃工時無法消化
- KOL 分批很重要：避免同時審核大量腳本，建議系統未來支援 KOL 排程衝突警示
